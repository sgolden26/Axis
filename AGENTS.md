# Project context

Axis is a hackathon entry for the SCSP Hackathon 2026 wargaming track. The aim is a modular wargaming platform that fuses a tactical layer (troop movement, kinetics, combat simulation) with a political layer (web-scraped sentiment, protest activity, leader pressure, troop morale) so that battlefield outcomes and strategic decisions react to live, public, unclassified signals. The visual language targets a Palantir/Maven feel: dark slate UI, hairline borders, mono-typography, faction-coded overlays on a real geographic map. The codebase is intentionally split into independent modules so two contributors can work in parallel and so the simulation engine and news/intel pipeline evolve without coupling.

## Stack

- Frontend: React + TypeScript (strict) + Tailwind + MapLibre GL + Zustand + Zod. Entry: `frontend/src/App.tsx`.
- Backend: Python 3.11+, OOP-first (ABC + factory + fluent builder), stdlib `dataclasses` for domain. Entry: `backend/axis/cli.py` (`python -m axis`).
- Contract: a single versioned `state.json` produced by the backend, validated by Zod on load. Schema lives in `docs/schema.md`.

## Repo map

- `backend/axis/domain/` immutable domain models (`Faction`, `Country` + nested dimension dataclasses, `City`, `Territory`, `Oblast`, `Frontline`, military_assets (`Depot`, `Airfield`, `NavalBase`, `BorderCrossing`, `SupplyLine`, `IsrCoverage`, `MissileRange`, `AreaOfResponsibility`), `Theater`, `Coordinate`, plus political layer in `political.py`: `LeaderSignal`, `LeaderSignalType`, `RegionPressure`, `FactionPressure`, `PressureState`, `GapEvent`, `CredibilityTrack`).
- `backend/axis/domain/countries/` `CountryRepository` ABC plus `StubCountryRepository` (hand-authored dossiers); plug-in point for future GDELT/Factbook/IISS adapters.
- `backend/axis/units/` `Unit` ABC plus `InfantryBrigade`, `ArmouredBrigade`, `AirWing`, `NavalTaskGroup`.
- `backend/axis/factories/` `UnitFactory` (factory method) and `ScenarioBuilder` (fluent builder, with `add_*` for every entity kind including oblasts, depots, airfields, naval bases, crossings, supply lines, ISR, missile arcs, AORs, frontlines, plus political: `with_current_turn`, `with_global_deadline`, `add_faction_pressure`, `add_region_pressure`, `add_credibility`, `add_leader_signal`).
- `backend/axis/scenarios/` concrete scenario seeds; v1 ships `eastern_europe` rebranded as a Russia/Ukraine theatre with seeded political state (`_add_political`).
- `backend/axis/serialization/` `SnapshotExporter` (Theater -> JSON, including the Phase 9 political fields).
- `backend/axis/sim/orders.py` polymorphic `Order` ABC, `OrderRegistry`, and `OrderBatch` (validate-all, then phased apply-all with seeded `BatchRng`). Concrete kinds: `MoveOrder`, `EntrenchOrder`, `EngageOrder`, `RebaseAirOrder`, `AirSortieOrder`, `NavalMoveOrder`, `NavalStrikeOrder`, `MissileStrikeOrder`, `ResupplyOrder`, `InterdictSupplyOrder`. New kinds register against the same registry without touching the HTTP layer.
- `backend/axis/sim/combat.py` stochastic-simple combat, range tables (`engagement_range_km`, `AIR_SORTIE_RADIUS_KM`, `AIR_REBASE_RADIUS_KM`, `NAVAL_MOVE_RADIUS_KM`), `haversine_km`, `is_target_detected` (range + ISR / self-detect), and the `PoliticalLedger` rollup that nudges `Country` opinion dimensions and `Oblast` control after kinetic outcomes.
- `backend/axis/sim/political_engine.py` post-batch sim hook: advances `current_turn`, computes `signal_severity` from the latest leader signal and `action_severity` from the executed batch, then runs `CredibilityEngine.record_action` and `PressureEngine.advance` so the political layer evolves on every successful execute.
- `backend/axis/server/` FastAPI app + `TheaterStore` singleton. `python -m axis serve` runs uvicorn; FE dev proxies `/api` to it. Endpoints: `GET /api/state`, `GET /api/signals` (political-layer slice), `POST /api/orders/execute` (single-team batch), `POST /api/orders/execute_round` (hot-seat: multiple per-team batches validated independently then applied in one phased pass with a shared `BatchRng`), `GET /api/orders/catalogue`, `POST /api/orders/suggest` (free-form intent → validated orders via OpenAI), `POST /api/reset`, `POST /api/decision/explain` (per-row narrative + sources for a `(action, region, team)` tuple; FE posts the live `RegionIntel` slice in the body so the explanation is deterministic with what the user sees), `POST /api/ai/design_scenario` (Phase 11 LLM-backed Scenario Designer, mode `initial`; display-only, never mutates the live theatre). The CLI `export` path remains unchanged for the static demo workflow.
- `backend/axis/decision/explain.py` builder behind `/api/decision/explain`. Re-runs the evaluator, then attaches a per-row `summary` (qualitative paragraph), `math` (formula), `sources[]` (intel events with URLs, leader signals with `source_url`, action-catalog references, scenario-assumption notes), and a structured `data` block for political rows (faction pressure with drivers, credibility track + recent gap history + recent leader signals from the issuer). Every `Action` carries an optional `wargame_note` 1-sentence string surfaced in the FactorFlowGraph hover and the explanation header.
- `backend/axis/intel/` morale pipeline plus the Phase 9 political modules: `credibility.py` (two-track bilateral updater, decay), `pressure.py` (faction/region intensity rollup with deadline ramp), `leader_statements.py` (`StubAdapter` reading `data/leader_statements.json`, `GDELTAdapter` shell against the public GDELT 2.0 Doc/Events API with CAMEO→`LeaderSignalType` fold).
- `backend/axis/ai/` LLM agent package. Two NATO-handbook-grounded system prompts: `director_prompt.md` (Wargame Director / Facilitator, *Execute* step, per-turn scripted injects + non-player leader signals + facilitator notes; prompt-only) and `scenario_designer_prompt.md` (Scenario Designer, *Design* step, three modes `initial` / `redesign` / `wrap_up`, emits a design brief plus a mode-shaped scenario JSON envelope). Phase 11 wires the designer to OpenAI via `scenario_designer.py` (`OpenAIScenarioDesigner`): loads the MD prompt verbatim, posts a single `response_format=json_object` chat completion, returns the parsed envelope, raises `ScenarioDesignerError` on failure. Reads `OPENAI_API_KEY` and (optional) `AXIS_AI_MODEL` env vars; default model `gpt-4o-mini`. Display-only: the endpoint never mutates the live theatre. Future siblings: `red_cell.py`, `leader.py`, `adjudicator.py`.
- `frontend/public/borders/` static Natural Earth GeoJSON: admin-0 primary (RU/UA/BY/PL/MD/RO/GE), admin-0 context ring, admin-1 Ukrainian oblasts + Crimea/Sevastopol.
- `frontend/src/map/` MapLibre wrapper, layer modules per kind (territory, cities, units, oblast, country, asset + dock badge, coverage, frontline, supply, missile range fill/line, `actionDraftOverlay` for in-progress action drafts, `stagedOrdersOverlay` for cart-staged orders, `strikeArcLayer` for the post-execute strike cinematic with curved trajectories, smoke trail, tracer + halo, origin flash, multi-ring hits, amber starburst on intercept, and per-arc stagger so dense rounds remain legible), `mapRef.ts` singleton, and `geo.ts` geospatial primitives. Click-stack cycling (repeated clicks within 6px advance through the resolved selectables) is wired in `MapView`'s click handler.
- `frontend/src/ui/` HUD, `PoliticalStrip` (always-visible turn + global/per-faction NATO Wargaming Handbook (HQ SACT, 2023, Fig. 7) team-goal pips: `{TEAM} TEAM GOAL · {goal} T-N · pressure {pct}%`), `LeftDock` (FilterChips with a `missile_ranges` chip + OOBTree), `RightPanel` (Prompt tab / Context tab / Decision tab in that order: `PromptPanel` posts to `/api/ai/design_scenario` with a sponsor problem statement + 4 designer knobs and renders the returned design brief and political seed inline; `Sidebar` for context; docked `DecisionEngine` plus full-screen `DecisionImmersive` with a client-built factor flow graph from `intel` + outcome estimate), CountryRoster (with outgoing-credibility chip column), Bottom bar (MeasureTool, Bookmarks, EventTicker), Minimap, HoverCard, KeyboardShortcuts, floating `AssistantBar`. Sidebar holds the per-kind detail panels for every selectable entity (city, unit, territory, country, oblast, depot, airfield, naval base, crossing, supply, ISR, missile, AOR, frontline). `UnitDetail` and `AssetDetails` (depot, missile) render concrete action buttons (move, engage, entrench, rebase, sortie, naval move/strike, missile strike, interdict, request resupply); shared primitives `ActionDraftPanel` and `ResupplyPicker` live in `Sidebar/primitives/`.
- `frontend/src/state/visibility.ts` perspective-aware helpers: `computeVisibleUnitIds(scenario, team, selectedUnitId?)` for basic FoW (friendly/neutral always visible; enemy visible only inside friendly ISR cones or within 60km of a friendly unit; the current selection is forced visible) and `computeDocking(scenario)` which returns the air/naval units sitting at a friendly airfield/naval base (via `home_base_id` co-location or 6km proximity to a compatible friendly asset). `MapView` filters the unit feature collection through both helpers so docked units hide under a faction-coloured count badge on their host asset and enemy units only appear when detected.
- `frontend/src/state/store.ts` Zustand store (selection, hover, visible layers, right tab, left/right panel open, `decisionImmersiveOpen`, OOB expansion, bookmarks, measure path, ticker pause, help overlay, choropleth metric, `playerTeam` red/blue for order gating, per-team `stagedOrders` cart, per-team `roundReady` lock for hot-seat staging, `cartOpen`, `executing`, `unitPositionOverrides` for the move/rebase/naval-move animator, `actionDraft` + `actionDraftToast` for map-pick draft flows, `replay` phase machine for the post-execute cinematic). `frontend/src/state/playerTeam.ts` exports `PlayerTeam` and `isFactionControllableByPlayerTeam` vs `Faction.allegiance`. `frontend/src/state/orders.ts` defines the `StagedOrder` discriminated union and DTO mapping; new order kinds extend the union and add a builder. `frontend/src/state/actionDraft.ts` defines the `ActionDraft` types, mirrored range tables, candidate enumeration, and frontend `isTargetDetected` parity for engage/sortie/naval-strike/missile-strike/rebase/naval-move/interdict drafts. `frontend/src/state/replay.ts` defines the `ReplayEvent` union (move | engage | strike) and `buildReplayEvents` which the store uses after a successful round to sequence Phase 1 (moves animator), Phase 2 (strike cinematic; duration auto-fits `planStrikes(events).totalDurationMs` from the `strikeArcLayer`, capped via stagger so rounds stay short, skippable), and Phase 3 (persistent damage chips with co-located events collapsing into a `×N` summary chip you can click to expand). The store's `executeRound` posts both teams' batches to `/api/orders/execute_round` only when both `roundReady` flags are set, then walks the phase machine; while a team is Ready, its staging mutators (start*Draft, stage*, removeStagedOrder, clearStagedOrders) become no-ops.
- `frontend/src/types/` Zod schemas mirroring the Python models (`scenario.ts`, `country.ts`, `oblast.ts`, `frontline.ts`, `military_assets.ts`, `orders.ts`, `political.ts`).
- `frontend/src/decision/politicalContext.ts` builds a `PoliticalContext` for the FE evaluator from the live snapshot (issuer faction = primary faction matching `playerTeam`, target = focused region's controller). The evaluator factors `pressure_aggression_bias` (scaled by issuer pressure) and `credibility_weight` (scaled by bilateral immediate credibility) into the probability and emits matching breakdown rows in `OutcomeCard`. Each `BreakdownItem` carries a stable `key` (`base`, `morale`, `trend`, `severity`, `cat:<category>`, `pressure`, `credibility`) the explain endpoint matches against.

In `DecisionImmersive`, the `OutcomeCard` rows become buttons: clicking a row expands an inline accordion with summary text, math, sources (clickable links for intel events and GDELT-sourced leader signals), and for political rows a structured panel with issuer pressure drivers, full bilateral credibility track including recent gap history, and recent leader signals from the issuer. - `frontend/src/api/designScenario.ts` POSTs the prompt-panel form (`sponsor_problem_statement` + `knobs`) to `/api/ai/design_scenario` and returns the `design_brief` + `scenario` envelope. `frontend/src/ui/Prompt/PromptPanel.tsx` holds the form state locally (free-text plus pressure profile / credibility profile / third-party intensity / deadline turn), renders the brief and political seed inline on success, and surfaces backend errors verbatim. - `frontend/src/ui/Orders/OrdersCart.tsx` top-right floating cart of staged orders for the active team. Footer hosts the hot-seat soft toggle: Ready / Unready per team, an at-a-glance "you · other team" status line, and a single `Execute round` button that lights up only when both teams are Ready. `frontend/src/api/executeRound.ts` POSTs `{batches: [OrderBatchDTO, OrderBatchDTO]}` to `/api/orders/execute_round` for the hot-seat round. `frontend/src/ui/Replay/PhasePill.tsx` is the floating top-centre pill that walks through "Phase N of 3 · Moves/Strikes/Damage reports" with a Skip button on the first two phases and a Dismiss button on Reports; `frontend/src/ui/Replay/DamageChips.tsx` renders persistent monospace damage chips anchored to lonlat via `map.project()` for every kinetic outcome of the round.
- `frontend/src/ui/Assistant/AssistantBar.tsx` floating bar that takes a free-form intent, calls `POST /api/orders/suggest` (OpenAI Chat Completions, JSON mode) via `frontend/src/api/suggestOrders.ts`, and drops validated orders straight into the active team's cart tagged `source: "llm"` with a green "AI" pill. ## Conventions

- Coordinates are always `[lon, lat]` (GeoJSON / MapLibre order).
- Normalised metrics (`strength`, `readiness`, `morale`, `control`, country dimension scalars) are floats in `[0, 1]`. Bilateral relation scores are signed in `[-1, +1]`.
- IDs are namespaced and stable: `city.*`, `unit.<faction>.*`, `terr.*`, `oblast.<iso>.*`, `depot.*`, `afld.*`, `nbase.*`, `cross.*`, `supply.*`, `isr.*`, `msl.*`, `aor.*`, `front.*`. Country ids are short ISO-style slugs (`ru`, `ua`, `by`).
- `country_id` on `City`, `Territory`, `Unit` is optional. Alliance-only entities omit it.
- Oblasts carry `iso_3166_2` (e.g. `UA-46`) for join with the admin-1 GeoJSON.
- `available_actions` on every entity (Unit, Country, Oblast, asset, etc.) is a list of affordance ids; v1 renders them as disabled buttons.
- The frontend must ignore unknown fields in `state.json` for forward compatibility.
- Bump `schema_version` in both Python and Zod mirrors in the same change for any breaking schema edit.
- Domain code never imports from `sim`/`intel`/`ai`; the dependency graph is one-way. Country data flows from a `CountryRepository` so adapters can swap without touching scenarios.
